# 支付截图 OCR 结构化解析设计文档

## 背景

当前 OCR 链路已经具备以下能力：

- 图片选择、拍照、预览流程稳定
- 原始文本识别已切换为 `ML Kit Chinese` 主识别，`Paddle OCR` 作为兜底
- 金额与日期在部分支付截图场景下已能正确提取

但在微信支付、美团支付详情这类 UI 截图中，商户提取仍然不稳定，容易出现以下问题：

- 将平台名识别为商户，例如 `美团`
- 将营销标签识别为商户，例如 `特价外卖团购`
- 将字段名误识别为商户，例如 `支付时闻`

根因不是单纯的旋转问题，而是当前链路丢失了 OCR 的布局信息。

## 当前问题定位

现有实现流程如下：

1. OCR 引擎返回 `textBlocks -> lines`
2. 仅保留每一行的纯文本
3. 使用换行拼接成 `rawText`
4. 解析器基于纯文本规则推断金额、日期、商户

该方案的问题是：

- 丢失了每一行文字的坐标信息
- 无法判断文字位于页面顶部、中部还是底部
- 无法判断字段名与字段值之间的对应关系
- 无法区分平台标签、营销文案、真实商户名

因此，当前解析器虽然能通过规则过滤一部分错误候选，但仍然属于“猜测式提取”，在结构复杂的支付截图中不够稳。

## 目标

新增一套“支付截图结构化解析”能力，使系统不仅识别文字，还能利用页面布局关系提取字段。

目标结果：

- 金额优先提取真实交易金额
- 日期优先提取支付时间对应值
- 商户优先提取真实店铺名，而不是平台名或营销文案

目标示例：

- 输入截图：微信/美团支付详情页
- 输出金额：`13.40`
- 输出日期：`2026年04月01日14:26:15`
- 输出商户：`鹿角巷(万象宜宾天地店)`

## 设计原则

### 1. 保留 OCR 行级布局信息

不再只保留纯文本，应保留每一行的几何信息：

- 文本内容
- left
- top
- right
- bottom
- centerX
- centerY

这样后续解析可以基于位置做判断，而不是只依赖关键词。

### 2. 区分“通用票据解析”和“支付截图解析”

当前 `OcrReceiptParser` 仍然保留，继续负责：

- 普通票据
- 简单小票
- OCR 结果较规整的文本场景

新增支付截图专用解析器负责：

- 微信支付详情
- 美团支付详情
- 饿了么订单支付详情
- 其他类似左侧标签、右侧值的 UI 截图

### 3. 优先做字段配对，而不是全局猜测

对于支付截图，优先寻找如下关系：

- `支付时间` -> 右侧或下一行的时间值
- `商品` -> 右侧或下一行的商品/店铺名
- `当前状态` -> 右侧或下一行的状态值

商户提取应优先从 `商品` 对应值中获得，而不是在整页所有文本中猜一个最像商户的字符串。

## 建议数据结构

建议新增行级 OCR 模型：

```kotlin
data class OcrLine(
    val text: String,
    val left: Int,
    val top: Int,
    val right: Int,
    val bottom: Int,
) {
    val centerX: Int get() = (left + right) / 2
    val centerY: Int get() = (top + bottom) / 2
    val width: Int get() = right - left
    val height: Int get() = bottom - top
}
```

建议新增结构化 OCR 结果：

```kotlin
data class OcrStructuredResult(
    val rawText: String,
    val lines: List<OcrLine>,
)
```

## 建议解析流程

### 阶段一：OCR 输出结构化

修改 `MlKitOcrEngine`，使其返回：

- 原始文本
- 行列表及坐标

此阶段不改变 UI，只是让数据层拥有足够的信息。

### 阶段二：支付截图版面分区

将页面粗略分成以下区域：

- 顶部导航区
- 顶部平台信息区
- 中部金额区
- 详情字段区
- 底部辅助功能区

其中，重点关注“详情字段区”。

可以先使用简单规则实现：

- 按 `top` 排序
- 依据 `centerY` 聚类相邻行
- 找到金额所在位置
- 重点分析金额附近及其下方的字段区域

### 阶段三：字段名和值配对

在详情字段区内查找标签：

- `支付时间`
- `商品`
- `收单机构`
- `支付方式`
- `交易单号`
- `商户单号`

配对规则建议：

- 若同一水平带上存在右侧文本，优先取右侧文本为值
- 若标签下方紧邻一行是长文本，也可视为该字段值
- 若同一字段值跨两行，可尝试合并相邻文本

### 阶段四：商户清洗

若商品字段值提取为：

- `鹿角巷(万象宜宾天地店)-美团App-`

则进行清洗：

- 去除平台后缀，例如 `-美团App-`
- 去除多余尾部符号
- 保留门店括号信息

最终得到：

- `鹿角巷(万象宜宾天地店)`

### 阶段五：回退策略

若支付截图专用解析失败，则回退到当前 `OcrReceiptParser`：

- 保证现有小票能力不受影响
- 保证新解析器失败时系统仍能给出可用结果

## 文件改动建议

建议按以下顺序改动：

1. `app/src/main/java/com/example/ocrjizhang/data/ocr/MlKitOcrEngine.kt`
   目标：返回结构化 OCR 行信息

2. `app/src/main/java/com/example/ocrjizhang/data/repository/OcrRepository.kt`
   目标：传递结构化识别结果，而不是只传纯文本

3. `app/src/main/java/com/example/ocrjizhang/utils/`
   目标：新增支付截图专用解析器，例如 `PaymentScreenshotParser.kt`

4. `app/src/test/java/com/example/ocrjizhang/utils/`
   目标：新增支付截图解析测试，覆盖微信/美团/饿了么等截图样本

## 实施顺序

建议按以下顺序落地：

1. 定义 `OcrLine` 和 `OcrStructuredResult`
2. 改造 `MlKitOcrEngine` 返回结构化结果
3. 保持当前 UI 不变，仅让 Repository 拿到坐标信息
4. 新增支付截图专用解析器
5. 先实现 `支付时间` 和 `商品` 的标签值配对
6. 将商户提取切换为“商品字段优先”
7. 最后保留当前通用解析器作为回退

## 验收标准

针对美团支付详情样例，满足以下条件：

- 原始识别文本可正常显示
- 金额提取结果为 `13.40`
- 日期提取结果为 `2026年04月01日14:26:15`
- 商户提取结果为 `鹿角巷(万象宜宾天地店)`
- 不再将以下文本误识别为商户：
  - `美团`
  - `特价外卖团购`
  - `支付时间`
  - `支付时闻`

## 风险与说明

### 1. 通用 OCR 无法完全避免单字误识别

例如 `支付时间` 被识别为 `支付时闻`，这是字符识别层误差，不可能完全靠规则消灭。

应对方式：

- 对常见字段名建立近似匹配或容错表
- 优先使用几何关系和字段上下文，而不是完全依赖字段文本完全匹配

### 2. 不同平台页面结构存在差异

微信支付、支付宝账单、美团支付页、饿了么订单页布局不完全一致。

应对方式：

- 先支持“左标签 + 右值 / 下值”的主流模式
- 后续根据测试样本增加适配规则

### 3. 解析器职责不要过大

不建议继续把所有规则都堆进当前 `OcrReceiptParser`。

应对方式：

- 保持 `OcrReceiptParser` 负责通用票据
- 新增支付截图专用解析器
- 通过路由逻辑选择最合适的解析路径

## 结论

后续优化方向应从“继续堆文本过滤规则”转向“保留 OCR 坐标信息并做字段配对”。  
对于支付截图场景，真正稳定的方式不是让系统在整页文本里猜商户，而是通过页面布局找到 `商品` 对应值，再清洗得到商户名。
