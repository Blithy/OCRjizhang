# 当前开发顺序与阶段状态

## 文档目的

这份文档用于覆盖当前阶段的实际执行顺序，避免继续依赖旧的、已与项目现状不完全一致的计划文档。

适用原则：

1. 后续开发默认以本文件中的顺序为准。
2. 每一阶段必须形成最小可演示闭环后，再进入下一阶段。
3. 每完成一个阶段，都需要同步到 GitHub。

## 当前项目主线状态

截至 2026-04-01，以下主线已经具备可演示闭环：

1. 用户登录与本地会话保持
2. 分类管理闭环
3. 交易新增、编辑、删除闭环
4. OCR 识别主链路
5. OCR 结果带入记账表单
6. 支付截图结构化解析

说明：

1. OCR 支付截图场景已经能正确提取金额、日期和真实商户。
2. 记账录入页已经支持 OCR 预填提示与保存收口。
3. 统计页已经接入真实数据聚合与图表渲染基础能力。

## 当前推荐开发顺序

从现在开始，后续优先级按下面顺序推进：

1. 阶段 S1：统计页验证与轻量收口
2. 阶段 S2：演示用本地后端同步
3. 阶段 S3：全流程联调与答辩演示优化

## 各阶段说明

### 阶段 S1：统计页验证与轻量收口

当前状态：

1. 已有 `StatisticsRepository`
2. 已有日 / 周 / 月聚合
3. 已有饼图和柱状图渲染
4. 已有统计计算测试

本阶段目标：

1. 确认统计页在真实交易数据下展示正常
2. 确认日 / 周 / 月切换与图表刷新稳定
3. 仅修复真实验证中发现的问题

执行策略：

1. 如果统计页基础能力已经稳定，则不重写、不扩展复杂需求
2. 只做演示必要的收口
3. 完成后立即进入同步阶段

当前判断：

统计页基础闭环已经具备，因此 S1 不再作为大开发阶段处理，而是视为“已完成基础能力，后续仅按需修补”。

### 阶段 S2：演示用本地后端同步

这是当前真正的下一优先级。

目标：

1. 让本地账号、分类、交易可以和本机 Spring Boot 后端完成演示级同步
2. 提供一个可手动触发的“立即同步”入口
3. 保证离线时不阻塞本地记账
4. 保证联机后可以演示一次 push / pull 闭环

建议执行顺序：

1. 先实现 `SyncRepository` 或等价同步调度层
2. 打通本地待同步队列读取
3. 打通 `SyncService` 的 push / pull 调用
4. 先做手动同步入口，不先做复杂自动同步
5. 优先处理交易和分类，同步用户资料只保留最小演示能力
6. 完成后在个人中心提供同步按钮与结果提示

阶段完成标准：

1. 本地新增一笔交易后，可手动同步到本地后端
2. 远端数据可被拉回并更新本地
3. 同步过程有明确成功 / 失败反馈
4. 不要求复杂冲突策略全面实现，但要能稳定演示

### 阶段 S3：全流程联调与答辩演示优化

目标：

1. 统一首页、统计页、OCR 页、交易页的演示节奏
2. 补足关键提示文案
3. 确保真机演示时路径清晰
4. 准备答辩展示顺序

重点内容：

1. 检查主链路是否存在重复点击、空态、异常提示问题
2. 优化个人中心中的同步说明
3. 优化首页摘要与最近记录展示
4. 补关键测试和演示说明
