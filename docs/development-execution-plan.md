# Android OCR 记账 App 开发执行文档

## 1. 文档目的

本文档用于约束后续实际开发顺序，确保开发过程按照固定阶段推进，而不是在多个模块之间来回切换。

后续开发必须遵循以下原则：

1. 每次只推进一个明确阶段。
2. 每个阶段开始前先确认输入条件。
3. 每个阶段结束前必须完成最小可运行闭环。
4. 每个阶段都要有验证方式。
5. 每次提交都必须与当前阶段目标对应，并同步推送到 GitHub。

## 2. 当前项目状态

### 2.1 已完成内容

当前仓库已经完成以下基础骨架能力：

1. Android 工程初始化与 GitHub 同步链路。
2. `MVVM + Hilt + Room + Retrofit + Navigation + DataStore` 基础接入。
3. 单 Activity + Bottom Navigation 主框架。
4. 本地数据库实体、DAO、数据库类骨架。
5. 远程接口 Service、请求模型、响应模型骨架。
6. 首页、统计、我的、分类、OCR、记账录入占位页面。

### 2.2 当前未完成内容

以下内容仍然是业务空壳或未实现状态：

1. 登录注册真实业务流程。
2. App 启动时的登录态判断与页面路由。
3. 分类管理真实增删改查。
4. 交易真实增删改查。
5. 统计聚合与图表。
6. OCR 识别真实流程。
7. 同步队列与远程同步逻辑。
8. 最终 UI 统一收口。

## 3. 总体开发顺序

后续开发固定采用以下顺序：

1. 阶段 A：全局主题与认证壳层收口。
2. 阶段 B：用户登录注册与本地会话保持。
3. 阶段 C：分类管理闭环。
4. 阶段 D：交易增删改查闭环。
5. 阶段 E：统计聚合与可视化。
6. 阶段 F：OCR 识别与记账填充。
7. 阶段 G：远程同步闭环。
8. 阶段 H：联调整理、测试补全与演示优化。

说明：

1. 认证先于分类和交易，因为后者依赖用户身份与会话。
2. 分类先于交易，因为交易录入依赖分类选择。
3. 本地交易先于统计，因为统计依赖真实本地数据。
4. OCR 先做本地识别与填充，再接同步，避免同时处理两条复杂链路。
5. 同步放在后段实现，因为它依赖登录、分类、交易和本地数据结构稳定。

## 4. 每阶段详细规划

## 4.1 阶段 A：全局主题与认证壳层收口

### 目标

将当前项目从“骨架页面集合”收口成“具备登录前后页面边界”的真实应用壳层，并统一全局 Material Design 3 主题基线。

### 输入条件

1. 主 Activity 与导航已可运行。
2. Hilt、Navigation、DataStore 已接入。

### 本阶段任务

1. 将全局配色调整为符合要求的 Material Design 3 生活类风格：
   - 不使用绿色。
   - 不使用天蓝色。
   - 以暖橙、珊瑚、暖米白、中性灰为主。
2. 增加启动路由页或启动判断逻辑。
3. 明确“登录前页面”和“登录后页面”的导航边界。
4. 约束底部导航只在登录后主页面显示。

### 产出物

1. 统一主题色与基础样式。
2. 启动页或启动路由逻辑。
3. 认证页面导航入口。

### 完成标准

1. App 启动后能进入启动判定流程。
2. 未登录不会直接进入主业务页。
3. 登录后才进入主页面容器。

### 验证方式

1. `assembleDebug`
2. 手动运行并验证路由跳转

## 4.2 阶段 B：用户登录注册与本地会话保持

### 目标

完成完整的“注册 -> 登录 -> 保存 token -> 启动自动识别登录状态 -> 退出登录”闭环。

### 输入条件

1. 阶段 A 已完成。
2. 认证相关接口模型已具备骨架。
3. `SessionManager` 已存在。

### 本阶段任务

1. 创建认证模块包结构：
   - `ui/auth`
   - `repository/auth`
   - 必要的 mapper / ui state
2. 实现认证仓库：
   - 登录
   - 注册
   - 退出登录
   - 获取当前会话
3. 实现登录页 UI：
   - 用户名
   - 密码
   - 登录按钮
   - 跳转注册页
4. 实现注册页 UI：
   - 用户名
   - 密码
   - 昵称
   - 邮箱
   - 手机号
   - 注册按钮
5. 实现登录与注册 ViewModel：
   - 输入校验
   - loading 状态
   - 成功跳转
   - 失败提示
6. 登录成功后保存：
   - token
   - userId
   - username
   - nickname
7. 登录成功后拉起主页面。
8. 退出登录时清理本地会话。

### 本阶段不做

1. 找回密码
2. refresh token
3. 复杂账号安全策略

### 产出物

1. 登录页
2. 注册页
3. 启动自动判定登录状态
4. 退出登录能力

### 完成标准

1. 能注册新用户。
2. 能登录成功并进入主页面。
3. 重新打开 App 时能根据本地 token 判断是否直接进入主页。
4. 退出后会回到登录页。

### 验证方式

1. `assembleDebug`
2. 模拟成功登录流程
3. 模拟退出流程
4. 单元测试覆盖输入校验和 session 保存逻辑

## 4.3 阶段 C：分类管理闭环

### 目标

完成默认分类初始化与自定义分类的本地增删改查闭环，并预留同步入口。

### 输入条件

1. 阶段 B 已完成，具备当前用户会话。
2. Room 分类表、DAO、实体已存在。

### 本阶段任务

1. 增加分类 repository。
2. 登录成功后初始化默认分类。
3. 实现分类列表页：
   - 收入分类
   - 支出分类
   - 默认分类标记
4. 实现新增分类。
5. 实现编辑分类。
6. 实现删除自定义分类。
7. 删除分类时将历史交易迁移到“未分类”。
8. 所有分类变更写入同步状态。

### 产出物

1. 分类列表页真实数据展示。
2. 分类新增编辑删除流程。
3. 默认分类初始化逻辑。

### 完成标准

1. 登录用户首次使用时具备默认分类。
2. 可新增和编辑自定义分类。
3. 删除分类逻辑符合规格要求。

### 验证方式

1. `assembleDebug`
2. 本地数据库实际写入验证
3. 分类迁移逻辑单元测试

## 4.4 阶段 D：交易增删改查闭环

### 目标

实现本地交易的完整录入、列表、编辑、删除闭环，让 App 具备真正可用的基础记账能力。

### 输入条件

1. 阶段 C 分类已可用。
2. 交易实体与 DAO 已存在。

### 本阶段任务

1. 创建交易 repository。
2. 将当前占位的交易录入页改成真实表单：
   - 收支类型
   - 金额
   - 分类
   - 日期
   - 商户
   - 备注
3. 实现交易列表展示。
4. 实现新增交易。
5. 实现编辑交易。
6. 实现删除交易。
7. 交易变更同步写入本地同步状态。
8. 首页最近交易区接入真实数据。

### 产出物

1. 交易录入表单页
2. 交易列表页
3. 交易新增、编辑、删除逻辑

### 完成标准

1. 可以真实录入一笔收入或支出。
2. 可以编辑已存在交易。
3. 可以删除交易。
4. 首页能看到最近交易摘要。

### 验证方式

1. `assembleDebug`
2. 本地数据库 CRUD 验证
3. ViewModel 单元测试

## 4.5 阶段 E：统计聚合与可视化

### 目标

基于本地交易数据完成日、周、月统计，并接入图表可视化。

### 输入条件

1. 阶段 D 已完成，真实交易数据可用。

### 本阶段任务

1. 实现统计 repository。
2. 实现按日、周、月聚合。
3. 统计总收入、总支出、结余。
4. 接入饼图展示支出分类占比。
5. 接入柱状图展示时间维度收支。
6. 首页显示简化统计摘要。

### 完成标准

1. 统计页可切换日、周、月。
2. 图表能根据真实本地数据变化刷新。

### 验证方式

1. `assembleDebug`
2. 统计工具单元测试
3. 页面联调验证

## 4.6 阶段 F：OCR 识别与记账填充

### 目标

实现“拍照或选图 -> OCR 识别 -> 解析金额日期商户 -> 带入记账页”的闭环。

### 输入条件

1. 阶段 D 已完成，交易录入表单真实存在。
2. 网络层与 OCR service 已存在。

### 本阶段任务

1. 接入 Photo Picker 或相册选图。
2. 接入拍照流程。
3. 实现图片压缩与 Base64 编码。
4. 实现百度 token 获取与缓存。
5. 实现票据识别调用。
6. 解析金额、日期、商户。
7. 将 OCR 结果带入记账录入页。
8. 保存 OCR 历史到本地。

### 完成标准

1. 可以选择图片并识别。
2. 至少能提取部分字段。
3. 可以基于识别结果保存交易。

### 验证方式

1. `assembleDebug`
2. 实机或模拟器选图验证
3. OCR 解析逻辑测试

## 4.7 阶段 G：远程同步闭环

### 目标

在登录、分类、交易已稳定的前提下，补齐远程同步逻辑。

### 输入条件

1. 阶段 B、C、D 已完成。
2. 本地同步队列表已可用。

### 本阶段任务

1. 实现登录后拉取远端数据。
2. 实现分类与交易的 push 逻辑。
3. 实现 pull 逻辑。
4. 实现 `updatedAt` 冲突合并。
5. 实现离线补同步。
6. 个人中心增加“立即同步”入口。

### 完成标准

1. 登录后可拉取用户云端数据。
2. 本地操作恢复联网后可同步。
3. 手动同步入口可用。

### 验证方式

1. `assembleDebug`
2. 本地 Spring Boot 联调
3. 同步队列验证

   - 手动新增交易

   - 统计查看
   - 本地同步
   - 退出登录
4. 只有当上述链路稳定后，才考虑额外优化项。

## 6. 当前立即要做的事

当前下一步固定为：

### Step H1

统一首页、个人中心、同步结果等关键位置的“演示版口径”文案，避免继续使用“开发中”表达。

### Step H2

补充一份独立的答辩演示顺序文档，明确老师演示时建议的页面顺序、讲解点和环境准备项。

### Step H3

在真机上按完整顺序复测登录、分类、交易、OCR、统计和同步，发现问题后小步修复并立即验证。

### Step H4

将“新增记账”从普通导航页面重构为真正覆盖在当前一级页面上的底部弹层：

1. 底层账本、资产、统计、我的页面保持存活。
2. 首页 FAB、最近记录编辑入口、OCR 带入入口统一改为弹出底部记账面板。
3. 关闭记账面板时只收起弹层，不重新创建底层页面。
4. 交易录入、编辑、OCR 预填继续复用现有交易业务逻辑，避免同时改动 UI 和数据层。

### Step H5

将资产页从占位结构升级为可演示的本地账户闭环：

1. 新增账户实体、DAO、仓库与数据库迁移。
2. 为当前用户初始化现金、微信、支付宝、银行卡等默认账户。
3. 资产页展示总资产、账户数量和资金账户列表。
4. 支持账户新增、编辑、删除和余额维护，为后续交易联动预留基础。

### Step H6

将“新增记账”和“资产账户”正式联动起来：

1. 在新增记账面板中加入账户选择能力。
2. 新增交易时明确本次收支归属到哪个资金账户。
3. 保存交易后自动更新对应账户余额。
4. 编辑和删除交易时同步回写账户余额，保证账本与资产一致。

### Step H7

收口新增记账面板中的分类表现：

1. 扩充默认分类，覆盖更常见的生活场景，降低首次上手成本。
2. 将当前“首字 + 文字”的分类样式升级为“图标 + 文字”。
3. 保持分类图标风格统一，符合当前 Material Design 视觉方向。
4. 评估分类项密度与可点击区域，避免图标改造后影响真机操作。
5. 分类管理页与新增记账面板保持同一套图标语义，避免用户在两个入口看到不同表达。
6. 新增与编辑自定义分类时支持手动选择图标，而不再只依赖名称自动推断。
7. 从新增记账进入分类管理时，返回应重新回到新增记账上下文，避免流程被打断到首页。

## 8. 偏离处理规则

若开发过程中出现新需求或阻塞，处理顺序如下：

1. 先判断是否影响当前阶段的输入条件。
2. 若影响当前阶段，则先更新本执行文档。
3. 若不影响当前阶段，则记录为后续事项，不中断当前阶段开发。

后续开发默认以本文档为执行准则。
