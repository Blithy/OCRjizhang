# 小记OCR记账 结构改版规范（2026-04-02）

## 1. 文档目的

本文档用于约束本项目下一阶段的结构改版开发，目标不是继续零散补功能，而是将现有演示版升级为更接近生活类记账软件的产品形态。

本轮改版重点解决以下问题：

1. 首页信息层级还不够清晰，手动记账入口不够高频。
2. 新增记账页仍偏“表单页”，不够像真实生活记账 App 的高频录入体验。
3. 分类、OCR、资产、统计之间的入口关系还不够合理。
4. 现有数据模型无法完整支撑“资金账户”和“资产趋势”。
5. 需要在保持 Material Design 设计语言的前提下，统一交互节奏和页面结构。

本文档确认后，后续开发应以本文档作为执行依据。

## 2. 本轮改版的产品目标

### 2.1 产品定位

产品仍然是生活类个人记账 App，不做金融工具或企业财务工具的气质表达。

整体体验要求：

1. 保持 Material Design 3 的原生规范感。
2. 视觉上继续沿用当前暖色、米白、中性色体系。
3. 不使用绿色主视觉，不使用天蓝色主视觉。
4. 操作路径应更贴近“打开就能记一笔”的生活记账习惯。

### 2.2 本轮改版后的核心主线

登录后，用户看到的是“账本首页”，然后围绕四条稳定主路径展开：

1. 账本：首页总览、最近记录、OCR 入口、演示状态。
2. 记账：右下角主浮动按钮进入高频速记页。
3. 资产：查看总资产、账户余额分布、管理资金账户。
4. 统计：按周/月/年/全部/范围查看统计和趋势。

“我的”保留为辅助页面，主要承接账号、同步、分类管理等低频操作。

## 3. 信息架构改版结论

### 3.1 底部导航

底部导航改为四项：

1. 账本
2. 资产
3. 统计
4. 我的

分类页面不再作为底部一级页面，也不再保留在首页快捷入口中。

### 3.2 登录后的首屏

登录成功后进入“账本”页，而不是其他页面。

这是合理的，因为：

1. 账本页能先展示“本月收支总览”，让用户立即理解当前状态。
2. 账本页能承接“最近记账”和“最近演示状态”，适合毕业设计现场讲解。
3. 账本页放置全局主浮动按钮后，手动记账会成为最明显的主操作。

### 3.3 全局主操作

新增记账不再作为首页卡片入口，而改为全局主浮动按钮 `FAB`，固定在屏幕右下角。

这样做的优点：

1. 符合生活类记账 App“随时记一笔”的使用习惯。
2. 手动记账优先级明显高于分类管理等低频功能。
3. 首页信息结构更简洁，不需要再把“新增记账”当作一张普通卡片。

### 3.4 OCR 双入口

OCR 保留两个入口，但不抢占 `FAB` 主操作位。

推荐放置方式：

1. 入口 A：新增记账页内，作为“从票据带入”的就近入口。
2. 入口 B：账本首页中部的“票据识别功能卡”。

不建议把第二个 OCR 入口放到顶部工具栏图标里，原因是：

1. 顶部栏图标展示力不够，无法突出这是亮点功能。
2. 首页功能卡更利于答辩时讲解“这是本软件的智能识别亮点”。
3. OCR 是高亮功能，但不是最高频功能，因此不应替代主 `FAB`。

## 4. 页面结构规范

## 4.1 账本页（首页）

### 4.1.1 页面目标

账本页负责同时承接“状态查看”和“快速进入高频功能”。

### 4.1.2 页面结构

页面从上到下建议为：

1. 顶部欢迎区或页面标题区。
2. 本月收支总览卡。
3. OCR 功能卡。
4. 最近记账列表。
5. 最近演示状态卡。
6. 右下角全局新增记账 `FAB`。

### 4.1.3 各区块说明

#### 本月收支总览卡

保留当前首页已有的“本月收入 / 本月支出 / 本月结余”结构，但改成更明确的三段式信息布局。

#### OCR 功能卡

作为首页最重要的次级功能入口，文案建议明确表达：

1. 可从相册或拍照识别支付截图 / 小票。
2. 识别结果可直接带入记账。

此卡片应包含：

1. 主标题
2. 一句解释文案
3. 一个主按钮，如“开始识别”

#### 最近记账

保留当前最近交易列表，但视觉上从“卡片里的说明文案”改成更轻量的列表容器，突出最近几笔数据。

#### 最近演示状态

保留现有“演示状态 / 同步状态”区块，但建议只保留对答辩有帮助的信息，例如：

1. 当前登录账号
2. 最近同步时间
3. OCR 最近一次识别状态

不建议让此区域过大，避免抢占首页的主要信息层级。

### 4.1.4 交互规则

1. 点击 `FAB` 进入新增记账页。
2. 点击最近记录可进入编辑页。
3. 点击 OCR 功能卡进入 OCR 页。
4. 首页不再提供“分类管理”快捷入口。

## 4.2 新增记账页

### 4.2.1 页面定位

新增记账页从“表单页”升级为“高频速记页”。

整体交互参考你提供的截图，但样式仍采用当前项目的 Material Design 3 语言，不直接照搬 iOS 式布局。

### 4.2.2 页面结构

页面从上到下建议为：

1. 顶部栏：关闭 / 保存 / 标题
2. 收支类型切换区
3. 分类选择区
4. 金额展示区
5. 时间与可选附加信息区
6. OCR / 图片辅助入口行
7. 自定义数字键盘区

### 4.2.3 交互规则

#### 收支类型

最上方使用分段控件显示：

1. 支出
2. 收入

如后续需要转账，再另开阶段，不在本轮强制加入。

#### 分类选择

分类改成图标 + 文字网格显示，不再使用下拉框作为主交互。

同一屏幕优先展示当前类型下的常用分类，建议首屏 8 到 12 个。

分类区底部提供一个轻量入口：

1. “管理分类”
2. “新增分类”

这两个入口可以合并成一个入口，点击后进入分类管理页并默认弹出新增能力。

#### 金额输入

金额改成大字号展示区，不再使用普通文本框作为主输入。

金额由页面底部的自定义数字键盘控制输入。

建议规则：

1. 默认显示 `¥0.00`
2. 支持数字、小数点、删除
3. 支持“保存再记一笔”
4. 支持“完成”

#### 时间信息

金额区下方保留时间选择入口，默认当前时间。

建议展示：

1. 日期
2. 时间

其中日期和时间可合并为一个按钮，点击后弹出日期时间选择器。

#### 附加信息

当前项目已有“商户”和“备注”，但不应继续放在主信息层的高位。

建议改为“次级信息条”或“展开更多信息”：

1. 商户
2. 备注
3. 资金账户

这样更符合生活记账的真实使用习惯，因为大多数用户录一笔账时，首先关心的是类别和金额，其次才是备注。

#### OCR 入口

新增记账页内部增加就近 OCR 入口，建议形式为：

1. 一个带图标的辅助操作 Chip
2. 或一个横向快捷行中的“票据识别”按钮

它的作用是让用户在“准备手动记账”时，随时切换成“从截图识别带入”。

### 4.2.4 本轮明确不保留的内容

1. 新增记账页底部不再展示交易记录列表。
2. 新增记账页不再承担“查看全部历史记录”的职责。

## 4.3 分类管理

### 4.3.1 分类入口调整

分类管理只保留两个入口：

1. 新增记账页中的“管理分类 / 新增分类”
2. “我的”页面中的“分类管理”

首页不再显示分类快捷卡。

### 4.3.2 默认分类增强

为了降低首次上手门槛，本轮需要补充一组更完整的默认分类，并为每个分类配置图标。

支出分类建议至少包含：

1. 餐饮
2. 购物
3. 交通
4. 日用
5. 娱乐
6. 住房
7. 医疗
8. 学习
9. 社交
10. 宠物
11. 旅行
12. 未分类

收入分类建议至少包含：

1. 工资
2. 奖金
3. 兼职
4. 理财
5. 报销
6. 转入
7. 其他收入
8. 未分类

图标策略建议：

1. 优先使用 Material Symbols Rounded。
2. 图标统一单色，避免过度花哨。
3. 分类图标和文字共同表达，不依赖图标单独识别。

### 4.3.3 分类编辑规则

1. 默认分类不可删除。
2. 自定义分类可新增、编辑、删除。
3. 删除分类时，历史交易迁移到同类型“未分类”。

## 4.4 OCR 模块

### 4.4.1 入口位置

本轮固定为双入口：

1. 首页 OCR 功能卡
2. 新增记账页辅助入口

### 4.4.2 页面职责

OCR 页继续保留：

1. 拍照
2. 相册选择
3. 图片预览
4. 开始识别
5. 识别结果确认
6. 带入记账

### 4.4.3 与新增记账页的关系

用户从 OCR 进入后，成功识别应跳转到新增记账页，并自动带入：

1. 金额
2. 日期时间
3. 商户
4. 默认分类猜测结果（若有）

若识别信息不完整，仍允许用户在新增记账页手动补完。

## 4.5 资产页

## 4.5.1 页面定位

资产页是本轮新增的一级页面，用于展示“我现在一共有多少钱、钱分别放在哪些账户”。

### 4.5.2 页面结构

建议页面从上到下为：

1. 总资产总览卡
2. 账户汇总信息卡
3. 资金账户列表
4. 新增账户按钮

### 4.5.3 资产口径建议

为了兼顾可演示性和实现复杂度，本轮资产模块采用“账户驱动”的简化方案，也就是把它明确做成“钱包/账户管理”，不扩展为更复杂的资产业务系统。

推荐账户类型：

1. 现金
2. 银行卡
3. 微信
4. 支付宝
5. 储蓄卡
6. 校园卡
7. 其他

推荐统计口径：

1. 总资产：全部资金账户当前余额总和
2. 账户数量：当前已启用账户数量
3. 最大账户：余额最高的账户
4. 最近变动：最近一笔影响账户余额的交易时间

这套设计的优点是：

1. 数据结构简单。
2. 页面上容易讲解。
3. 不需要扩展额外的复杂资产业务规则。

### 4.5.4 账户管理规则

本轮需要支持：

1. 新增账户
2. 编辑账户名称、类型、初始余额
3. 删除账户
4. 选择账户图标

建议默认提供一组常见账户：

1. 现金钱包
2. 微信钱包
3. 支付宝
4. 银行卡

### 4.5.5 与交易的关系

若要让资产模块真正成立，交易必须能够归属到某个资金账户。

因此本轮建议把“资金账户”作为新增记账页中的可选但强推荐字段。

推荐规则：

1. 收入记到某个账户，账户余额增加。
2. 支出记到某个账户，账户余额减少。
3. 编辑或删除交易时，需要同步回算账户余额。

## 4.6 统计页

### 4.6.1 统计筛选

统计页顶部筛选改为：

1. 周
2. 月
3. 年
4. 全部
5. 范围

默认进入时显示“月”。

### 4.6.2 周期切换条

筛选条下面增加周期导航条。

示例：

1. 当前模式为月时，显示“2026 年 4 月”
2. 左箭头切换到上一个月
3. 右箭头切换到下一个月

不同模式下的导航规则：

1. 周：按周前后切换
2. 月：按月前后切换
3. 年：按年前后切换
4. 全部：不显示前后切换
5. 范围：点击后弹出日期范围选择

### 4.6.3 页面结构

统计页从上到下建议为：

1. 统计模式切换条
2. 周期导航条
3. 收支概览卡
4. 收支统计柱状图
5. 支出分类饼图
6. 资产趋势图

### 4.6.4 数据口径

收支概览卡建议包含：

1. 支出
2. 收入
3. 结余
4. 日均支出或期均支出

图表定义建议：

1. 柱状图：当前周期内收入 / 支出对比
2. 饼图：当前周期内支出分类占比
3. 资产趋势图：净资产或总资产按时间变化

### 4.6.5 资产趋势图的实现前提

要做资产趋势图，单纯依赖当前账户余额是不够的，因此必须增加以下前提之一：

1. 交易记录绑定账户，并按历史交易回算资产变化
2. 或维护每日账户快照

本轮推荐优先采用方案 1，因为它与新增记账和资产模块天然联动，实现成本更低。

## 4.7 我的页

“我的”页面不做大改，但需要承担以下低频管理功能：

1. 账号信息
2. 立即同步
3. 分类管理
4. 演示环境说明或版本信息
5. 退出登录

这样能保持“我的”页简洁，同时不把低频管理入口挤进首页。

## 5. 数据模型增量规范

## 5.1 Category 增量字段

分类需要补充以下展示属性：

1. `iconResName` 或 `iconKey`
2. `sortOrder`
3. `isPinned`（可选，若需要常用分类排序）

## 5.2 Transaction 增量字段

交易模型建议新增：

1. `accountId`
2. `accountName`

这样交易才能真正归属到某个资金账户，并支撑资产统计。

## 5.3 AssetAccount 新实体

新增资金账户实体，建议字段如下：

1. `id`
2. `userId`
3. `name`
4. `type`
5. `iconKey`
6. `openingBalanceFen`
7. `currentBalanceFen`
8. `createdAt`
9. `updatedAt`
10. `syncStatus`

## 5.4 是否需要资产快照表

若仅做第一版演示，可以先不单独引入资产快照表。

第一版先通过“账户初始余额 + 历史交易”回算资产趋势。

只有当后续发现性能或精度问题时，再引入 `asset_account_snapshot` 表。

## 6. 导航与交互规范

### 6.1 底部导航显隐

以下页面显示底部导航：

1. 账本
2. 资产
3. 统计
4. 我的

以下页面隐藏底部导航：

1. 登录
2. 注册
3. 启动页
4. 新增记账页
5. OCR 页
6. 分类管理页

### 6.2 新增记账页的进入方式

建议支持以下进入方式：

1. 首页 `FAB`
2. 首页 OCR 识别完成带入
3. 编辑一条已有交易

### 6.3 动效要求

动效继续遵循“接近原生 Android / Material”原则：

1. 页面进入退出使用中短时长转场
2. `FAB` 弹入使用轻量缩放与位移
3. 账本页滚动时，`FAB` 可轻微隐藏或缩小
4. 不增加夸张弹跳和复杂共享元素

## 7. 可用性结论与建议

## 7.1 你的逻辑整体是清晰的

这次需求不是简单换皮，而是很明确地把产品重心从“演示很多模块”转成“像一个真实可用的生活记账 App”。

整体逻辑是成立的，尤其是以下几点非常正确：

1. 新增记账改成全局 `FAB`
2. 分类降为低频管理能力
3. OCR 作为亮点功能但不抢主操作位
4. 资产升级为一级页面
5. 统计页增加更完整的时间维度

## 7.2 需要提前确认的唯一关键点

资产功能一旦加入，交易和账户就必须建立关联，否则：

1. 账户余额无法自动变化
2. 资产趋势图无法成立
3. 资产页会变成静态展示页

因此本轮开发时必须把“账户”正式纳入交易模型，而不是只做一个孤立页面。

## 7.3 对用户上手是否友好

整体会比当前版本更好上手，原因是：

1. 用户一进来就能看到首页总览和最近记录。
2. 真正高频的“记一笔”变成右下角主按钮，学习成本更低。
3. 分类图标化后，第一次使用不需要先理解“分类管理页”。
4. OCR 放在首页卡片和记账页内，既容易发现，也不显得杂乱。

但为了保持上手简单，建议注意两件事：

1. 新增记账页首屏只展示核心字段，商户、备注、账户放进次级区。
2. 资产页先做“账户管理 + 汇总”，不要在第一版扩展成理财或其他复杂资产模块。

## 8. 开发执行顺序

本轮推荐严格按以下顺序推进：

1. 阶段 R1：更新执行文档与页面信息架构，确认底部导航重构方案。
2. 阶段 R2：重构首页与全局 `FAB`，完成“账本”页新版骨架。
3. 阶段 R3：重构新增记账页，替换成分类网格 + 金额面板 + 自定义数字键盘。
4. 阶段 R4：扩充默认分类和图标体系，收拢分类入口到新增记账页和“我的”页。
5. 阶段 R5：新增资金账户数据模型、本地数据库迁移和资产页。
6. 阶段 R6：将交易正式绑定账户，并补齐账户余额联动逻辑。
7. 阶段 R7：重构统计页时间筛选、周期导航、柱状图、饼图和资产趋势图。
8. 阶段 R8：收口 OCR 双入口、页面文案、动效和真机联调。

### 8.1 为什么这样排序

这样排序是为了避免返工：

1. 先改导航和页面骨架，再改页面内部交互。
2. 先把记账页确定下来，再让资产和统计依赖新的交易结构。
3. 先落地账户模型，再做资产趋势图，否则统计页后面还要重写一次。

## 9. 本轮暂不做的内容

为了控制复杂度，本轮先不做以下能力：

1. 转账独立业务流
2. 多账本
3. 分类颜色自定义
4. 复杂筛选器
5. 独立复杂资产系统
6. 账户间转移自动分录

## 10. 当前结论

如果按本文档执行，这一轮改版后的产品会从“功能齐全的演示版”升级为“结构更接近真实生活记账软件的演示版”，并且不会偏离我们原本的 Material Design 设计方向。

当前建议是：

1. 先按本文档确认结构。
2. 确认后从 `R2 首页 + FAB` 和 `R3 新增记账页重构` 开始做第一批实现。
3. 资产和统计放在第二批，因为它们依赖新的交易与账户模型。
